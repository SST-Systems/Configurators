# Zenject for Configurators

[English](README.md) | **Русский**

Этот пример показывает, как подключить Configurators к DI-контейнеру **Zenject** вместо связки `ServiceLocator` +
`Bootstrap`, которую используют остальные семплы. Выгода двойная: четыре менеджера становятся синглтонами
контейнера, которые можно `[Inject]`-ить где угодно, и — что важнее — твои хендлеры создаёт *сам контейнер*. А
значит, хендлер может `[Inject]`-ить собственные зависимости (аудиошину, сервис сейвов, клиент аналитики) вместо
обращения к статическому локатору.

Вся связка — это всего два небольших куска: фабрика, которая просит Zenject инстанцировать хендлеры, и инсталлер,
который связывает эту фабрику с менеджерами. Добавь инсталлер в Scene- (или Project-) Context — и всё, что связано с
Configurators, начинает резолвиться через DI.

## Что внутри

- `ZenjectHandlerFactory` — реализует `IHandlerFactory`, делегируя создание в `IInstantiator` из Zenject, поэтому
  каждый хендлер строится через контейнер и получает полный инжект.
- `ConfiguratorsZenjectInstaller` — `MonoInstaller`, который биндит `IHandlerFactory` → `ZenjectHandlerFactory` и
  четыре менеджера (`IInstructionManager`, `IModificationManager`, `IConditionManager`, `IExtensionManager`) как
  синглтоны.

## Как использовать

1. Убедись, что **Zenject/Extenject** есть в проекте (asmdef ссылается на `Zenject`).
2. Добавь в сцену **SceneContext** (или ProjectContext).
3. Добавь `ConfiguratorsZenjectInstaller` в список **Mono Installers** этого контекста.
4. Готово — в инжектируемом коде можно `[Inject]`-ить любой из менеджеров, а каждый хендлер, который используют твои
   конфигураторы, инстанцируется контейнером и может `[Inject]`-ить свои зависимости.

> Требует пакет **Zenject/Extenject**. Этот инсталлер заменяет связку `Bootstrap` + `ServiceLocator` из остальных
> семплов — выбирай один подход на сцену, не оба сразу.
